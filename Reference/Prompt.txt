  ## Project: SSIP Meal Order Web App (React + Vercel)                                                                                                                                                                             
   
  ### Goal                                                                                                                                                                                                                         
  Build a React web app that replicates the functionality of an existing Android meal ordering app
  (SSIPMealOrderAppII). The web app targets iPhone users (internal use only) and must be deployable
  on Vercel's free tier.

  ---

  ### Backend Context
  The backend is a WooCommerce site at: https://ssip-cafeteria.whew.life
  It is NOT a REST API — it is a traditional server-rendered WooCommerce site. All data is obtained
  by:
  1. Submitting HTML forms (login, checkout)
  2. Scraping HTML responses with a parser (like cheerio)

  The Android app uses OkHttp + JSoup to do this. The web app will use Vercel serverless functions
  + cheerio to do the same thing server-side (to bypass browser CORS restrictions).

  ---

  ### Architecture

  **Frontend**: React (Vite), hosted on Vercel
  **Proxy layer**: Vercel Serverless Functions (`/api/*`) — sits between React and WooCommerce
  **No database needed**: All state lives in the user's browser localStorage
  **No Firebase needed**: Vercel alone handles frontend + proxy

  #### Login Flow
  1. User enters WooCommerce username + password in React app
  2. React POSTs to `/api/login`
  3. Proxy does GET `/my-account/` on WooCommerce → extracts hidden nonce/form fields from HTML
  4. Proxy does POST `/my-account/` with username, password, and nonce fields
  5. WooCommerce responds with session cookies (e.g., `wordpress_logged_in_*`, `woocommerce_*`)
  6. Proxy returns those cookies as JSON to the browser
  7. Browser stores cookies in localStorage under key `wc_session_cookies`
  8. Every subsequent API call sends `{ cookies }` to the proxy, which attaches them to WooCommerce requests

  **Multi-user isolation**: Each user's browser holds their own WooCommerce session cookies in
  localStorage. The proxy is completely stateless — it stores nothing.

  ---

  ### Vercel Proxy Functions to Build

  All functions receive cookies from the request body and attach them to WooCommerce requests.
  Use `node-fetch` or native `fetch` + `cheerio` for HTML parsing in each function.

  | Route | Method | Purpose | WooCommerce URL |
  |---|---|---|---|
  | `/api/login` | POST | Authenticate, return session cookies | POST `/my-account/` |
  | `/api/meals` | POST | Get meals for a date, return parsed list | GET `/lunch/?menu-date=YYYY-MM-DD` |
  | `/api/orders` | POST | Get order history (paginated) | GET `/orders/` |
  | `/api/dashboard` | POST | Get wallet balance + upcoming meals | GET `/woo-wallet/` + `/orders/` |
  | `/api/place-order` | POST | Place a single order | POST `/?wc-ajax=checkout` |
  | `/api/cancel-order` | POST | Cancel an order | POST to cancel URL |

  #### Request shape for all authenticated routes:
  ```json
  { "cookies": { "cookie_name": "cookie_value", ... }, ...otherParams }

  How to attach cookies to WooCommerce fetch:

  const cookieHeader = Object.entries(cookies)
    .map(([k, v]) => `${k}=${v}`)
    .join('; ');
  fetch(url, { headers: { Cookie: cookieHeader } });

  Order Placement Detail (Critical)

  The Android app places orders via WooCommerce checkout. The flow is:

  1. GET /lunch/?menu-date=YYYY-MM-DD → scrape meal list (product IDs, prices, names, stall)
  2. For each meal to order:
  a. POST to add-to-cart endpoint with product_id and quantity=1
  b. GET /checkout/ → scrape hidden fields: _wpnonce, payment_method, billing fields, etc.
  c. POST /?wc-ajax=checkout with all checkout form fields populated with user's saved
     order details (first name, last name, phone, email, delivery time as order notes)
  d. Parse response JSON for result: "success" and the order ID
  3. Extract order_id and the cancel URL from the success response

  User order details (stored in localStorage):

  - First name, last name, phone number, email
  - Preferred delivery time (used as order notes)

  Key Implementation Notes

  1. Rate limiting: Add 500ms delay between batch order placements (match Android app behavior
  to avoid hitting WooCommerce rate limits)
  2. Session expiry: If any proxy call returns a redirect to /my-account/ (login page),
  treat it as session expired → clear cookies → redirect user to login
  3. Cookie extraction: After login POST, extract Set-Cookie headers from the WooCommerce
  response. In Node.js fetch, use response.headers.getSetCookie() or parse the raw
  set-cookie header array.
  4. CSRF for checkout: The _wpnonce in the checkout form expires quickly. Always fetch
  /checkout/ fresh immediately before placing each order — do not cache it.
  5. Mobile-first UI: Design for iPhone Safari. Use min-h-dvh not min-h-screen.
  Add <meta name="viewport" content="width=device-width, initial-scale=1">.
  6. PWA manifest (optional): Add manifest.json and service worker so iPhone users can
  add to home screen for an app-like experience.
